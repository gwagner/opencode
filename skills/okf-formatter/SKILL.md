## Open Knowledge Format Spec

OKF is an open, human- and agent-friendly format for representing knowledge: the metadata, context, and curated insight around data, systems, assets, processes, and concepts.

An OKF bundle is a directory of UTF-8 markdown files with YAML frontmatter. It is designed to be authored by humans, generated by agents, exchanged across organizations, and consumed by both. OKF requires no registry, central authority, database, SDK, or special tooling. If a file can be read as markdown and parsed for YAML frontmatter, it can be consumed as OKF.

### 1. Purpose

OKF standardizes only the minimal structure needed to make a knowledge corpus self-describing.

Goals:

1. Provide a universal format enrichment agents can write.
2. Tell consumption agents how to read, traverse, and present knowledge.
3. Enable exchange of knowledge across tools, systems, and organizations.
4. Require only the fields necessary for meaningful consumption.

Non-goals:

* Defining a fixed taxonomy of concept types.
* Prescribing storage, serving, indexing, or query infrastructure.
* Replacing schemas such as Avro, Protobuf, OpenAPI, database schemas, or catalog formats. OKF may reference them, but does not subsume them.

OKF content should be:

* Human-readable without tooling.
* Agent-parseable without bespoke SDKs.
* Diffable in version control.
* Portable across tools, organizations, and time.

### 2. Core Terms

* **Knowledge Bundle**: A self-contained directory tree of knowledge documents. This is the unit of distribution.
* **Concept**: One markdown document representing one unit of knowledge. It may describe a concrete asset such as a table or API, an abstract concept such as a metric or process, or anything between.
* **Concept ID**: The concept file path relative to the bundle root, without the `.md` suffix. Example: `tables/users.md` has concept ID `tables/users`.
* **Frontmatter**: YAML metadata block at the top of a markdown file, delimited by `---`.
* **Body**: Markdown content after the frontmatter.
* **Link**: A markdown link from one concept to another, used to express relationships beyond directory hierarchy.
* **Citation**: A link from a concept to an external or internal source supporting a claim in the body.

### 3. Bundle Structure

A bundle is a directory tree of markdown files:

```text
bundle/
├── index.md
├── log.md
├── <concept>.md
└── <directory>/
    ├── index.md
    ├── <concept>.md
    └── <subdirectory>/
        └── ...
```

A bundle may be distributed as:

* A git repository, recommended for history, attribution, and diffs.
* A tarball or zip archive.
* A subdirectory inside a larger repository.

Directory organization is domain-specific and left to the producer.

#### Reserved Filenames

The following filenames have special meaning at any level of the hierarchy and must not be used as concept documents:

| Filename   | Meaning                                                |
| ---------- | ------------------------------------------------------ |
| `index.md` | Optional directory listing for progressive disclosure. |
| `log.md`   | Optional chronological update history.                 |

All other `.md` files are concept documents.

Tags are represented through the `tags` frontmatter field. OKF does not define a separate tag index format; consumers may synthesize tag views by scanning frontmatter.

### 4. Concept Documents

Every concept document is a UTF-8 markdown file with:

1. YAML frontmatter delimited by `---`.
2. A markdown body.

#### Frontmatter

```yaml
---
type: <Type name>                  # REQUIRED
title: <Optional display name>
description: <Optional one-line summary>
resource: <Optional canonical URI for the underlying asset>
tags: [<tag>, <tag>, ...]
timestamp: <ISO 8601 datetime>
# producer-defined fields are allowed
---
```

Required:

* `type`: Non-empty string identifying the concept kind. Consumers use it for routing, filtering, and presentation.

Example type values:

* `BigQuery Table`
* `BigQuery Dataset`
* `API Endpoint`
* `Metric`
* `Playbook`
* `Reference`

Type values are not centrally registered. Producers should choose descriptive, self-explanatory values. Consumers must tolerate unknown types, usually by treating them as generic concepts.

Recommended fields, in priority order:

1. `title`: Human-readable display name. If omitted, consumers may derive one from the filename.
2. `description`: One-sentence summary for indexes, snippets, and previews.
3. `resource`: Canonical URI for the underlying asset, if the concept describes a concrete resource.
4. `tags`: YAML list of short strings for cross-cutting categorization.
5. `timestamp`: ISO 8601 datetime of the last meaningful change.

Extensions:

* Producers may add arbitrary frontmatter keys.
* Consumers should preserve unknown keys when round-tripping.
* Consumers should not reject documents because of unrecognized fields.

#### Body

The body is standard markdown. Producers should favor structured markdown, such as headings, lists, tables, and fenced code blocks, because structure improves human readability and agent retrieval.

No body sections are required. These headings have conventional meaning and should be used when applicable:

| Heading       | Meaning                                               |
| ------------- | ----------------------------------------------------- |
| `# Schema`    | Structured description of columns, fields, or shape.  |
| `# Examples`  | Concrete usage examples, often in fenced code blocks. |
| `# Citations` | Sources supporting claims in the body.                |

### 5. Examples

#### Resource-Bound Concept

```markdown
---
type: BigQuery Table
title: Customer Orders
description: One row per completed customer order across all channels.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders, revenue]
timestamp: 2026-05-28T14:30:00Z
---

# Schema

| Column | Type | Description |
|---|---|---|
| `order_id` | STRING | Globally unique order identifier. |
| `customer_id` | STRING | Foreign key into [customers](/tables/customers.md). |
| `total_usd` | NUMERIC | Order total in US dollars. |
| `placed_at` | TIMESTAMP | When the customer submitted the order. |

# Joins

Joined with [customers](/tables/customers.md) on `customer_id`.

# Citations

[1] [BigQuery table schema](https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders)
```

#### Concept Not Bound to a Resource

```markdown
---
type: Playbook
title: Incident response — data freshness alert
description: Steps to triage a freshness alert on the orders pipeline.
tags: [oncall, incident]
timestamp: 2026-04-12T09:00:00Z
---

# Trigger

A freshness alert fires when `orders` lags more than 30 minutes behind its expected SLA. See the [orders table](/tables/orders.md).

# Steps

1. Check the [ingestion job dashboard](https://example.com/dash).
2. Continue triage according to local operating procedure.
```

### 6. Cross-Linking

Concepts may link to other concepts using standard markdown links.

Recommended form: absolute bundle-relative links beginning with `/`.

```markdown
See the [customers table](/tables/customers.md).
```

Relative links are also allowed.

```markdown
See the [neighboring concept](./other.md).
```

A link from concept A to concept B asserts that a relationship exists. The specific relationship type, such as references, depends-on, joins-with, parent-child, or related-to, is conveyed by surrounding prose rather than by the link syntax.

Consumers that build graph views typically treat links as directed, untyped edges.

Consumers must tolerate broken links. A missing target is not malformed; it may represent incomplete or not-yet-written knowledge.

### 7. Index Files

An `index.md` file may appear in any directory, including the bundle root. It supports progressive disclosure by listing what is available before a human or agent opens individual documents.

Normally, index files contain no frontmatter. The only exception is the bundle-root `index.md`, which may include `okf_version`.

Index body format:

```markdown
# Section / Group Heading

* [Title 1](relative-url-1) - short description of item 1
* [Title 2](relative-url-2) - short description of item 2

# Another Section

* [Subdirectory](subdir/) - short description of the subdirectory
```

Entries should include descriptions from linked concept frontmatter. Producers may generate indexes automatically. Consumers may synthesize an index when none exists.

### 8. Log Files

A `log.md` file may appear at any hierarchy level to record change history for that scope.

Format:

* Flat list of date-grouped entries.
* Newest first.
* Date headings must use ISO 8601 `YYYY-MM-DD`.
* Entries are prose.
* Leading bold labels such as `**Update**`, `**Creation**`, or `**Deprecation**` are conventional, not required.

Example:

```markdown
# Directory Update Log

## 2026-05-22
* **Update**: Added new BigQuery table reference for [Customer Metrics](/tables/customer-metrics.md).
* **Creation**: Established the [Dataplex Playbook](/playbooks/dataplex.md).

## 2026-05-15
* **Initialization**: Created foundational directory structure.
* **Update**: Added progressive-disclosure guidelines to the root [index](/index.md).
```

### 9. Citations

When a concept body makes claims sourced from external material, sources should be listed under `# Citations` at the bottom of the document.

```markdown
# Citations

[1] [BigQuery public dataset announcement](https://cloud.google.com/blog/products/data-analytics/...)
[2] [Internal data quality runbook](https://wiki.acme.internal/data/quality)
```

Citation links may be:

* Absolute URLs.
* Bundle-relative paths.
* Paths into a `references/` subdirectory that mirrors external material as first-class OKF concepts.

### 10. Conformance

A bundle conforms to OKF v0.1 if:

1. Every non-reserved `.md` file contains parseable YAML frontmatter.
2. Every concept frontmatter block contains a non-empty `type` field.
3. Every `index.md` and `log.md`, when present, follows the structures above.

Consumers should treat all other rules as soft guidance.

Consumers must not reject a bundle because of:

* Missing optional frontmatter fields.
* Unknown `type` values.
* Unknown additional frontmatter keys.
* Broken cross-links.
* Missing `index.md` files.

This permissive model is intentional. OKF should remain useful as bundles grow, are refactored, or are partially generated by agents.

### 11. Relationship to Other Formats

OKF resembles:

* LLM-oriented markdown knowledge repositories.
* Personal knowledge tools such as Obsidian and Notion.
* Metadata-as-code approaches that store catalog knowledge near source code.

OKF differs by specifying the minimal interoperability rules without dictating tooling.

### 12. Versioning

This specification is OKF version `0.1`.

Version format: `<major>.<minor>`.

* Minor versions introduce backward-compatible additions, such as optional fields or conventional headings.
* Major versions may introduce breaking changes, such as renamed required fields or changed reserved filenames.

A bundle may declare its target version by adding this frontmatter to the bundle-root `index.md`:

```yaml
---
okf_version: "0.1"
---
```

This is the only case where frontmatter is allowed in an `index.md`.

Consumers that do not understand the declared version should attempt best-effort consumption rather than refusing the bundle.

### 13. Minimal Bundle Example

```text
my_bundle/
├── index.md
├── datasets/
│   ├── index.md
│   └── sales.md
└── tables/
    ├── index.md
    ├── orders.md
    └── customers.md
```

`datasets/sales.md`:

```markdown
---
type: BigQuery Dataset
title: Sales
description: All sales-related tables for the retail business.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales
tags: [sales]
timestamp: 2026-05-28T00:00:00Z
---

The sales dataset contains transactional tables, including [orders](/tables/orders.md) and [customers](/tables/customers.md).
```

`tables/orders.md`:

```markdown
---
type: BigQuery Table
title: Orders
description: One row per completed customer order.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders]
timestamp: 2026-05-28T00:00:00Z
---

# Schema

| Column | Type | Description |
|---|---|---|
| `order_id` | STRING | Unique order identifier. |
| `customer_id` | STRING | FK to [customers](/tables/customers.md). |
| `total_usd` | NUMERIC | Order total in USD. |

Part of the [sales dataset](/datasets/sales.md).
```
